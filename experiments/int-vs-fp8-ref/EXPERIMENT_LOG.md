# Experiment log: int-vs-fp8-ref
Started 2026-05-13. See experiments/int-vs-fp8-ref/plan.md.

- 2026-05-13: First measurement of the int-cast student against the actual fp8 production teacher (prior int-cast experiments all eval'd against fp32-dequant references). Built thin wrapper around `train_emulate.build_models` with `teacher_source=published`, `init_from_teacher=True`, `int_nonmatmul_bitexact=False`. Lambda H100 SXM5 (vast out of credit), us-south-2. First Lambda launch used the `macbook 2025` ssh key from CLAUDE.md which isn't on this box; relaunch with the local `d6-rollout` key fixed it. Also bumped Pillow on the box (PIL.Image.Resampling was missing in the base ubuntu image, blocking transformers import).
- 2026-05-13: Naive run (uniform int24 act, b=24, IntEmbedding, int non-matmul ops) → Qwen2.5-0.5B top-1 = 0.9128 vs fp8 teacher. Two diagnostic findings: (1) `init_from_teacher` per-row branch only copied 1 of 169 weight tensors because newer transformers loads fp8 weights into plain `nn.Linear` with `weight.dtype=float8_e4m3fn + weight_scale` rather than `CompressedLinear`; fixed by adding a check on `hasattr(tm, "weight_scale") and tm.weight.dtype == float8_e4m3fn`. After fix → 0.9377. (2) The int non-matmul ops (int RMSNorm/SiLU/softmax) account for ~6pp of the residual gap.
- 2026-05-13: **Headline:** with `activation_scheme=fp8_e4m3` (per-token fp8 e4m3 quant in the student matching teacher's fp8-dynamic exactly) + per-row fp8 weights cast from teacher + bf16 F.linear matmul + **teacher-equivalent float non-matmul ops** (`--no-patch-nonmatmul`), all three models hit bit-exact top-1 against the fp8 production teacher:
  - Qwen2.5-0.5B (RedHatAI fp8 per-row): **top1=1.0000, margin=0, KL=0** on 15,348 positions.
  - Llama-3.1-8B-Instruct (RedHatAI fp8 per-row): **top1=1.0000, margin=0, KL=0** on 14,984 positions.
  - Qwen3-8B (Qwen native block-fp8, requires `--activation-scheme block_fp8_e4m3 --block-fp8-kernel-path` to invoke Triton `w8a8_block_fp8_matmul_triton`): **top1=1.0000, margin=0, KL=0** on 15,348 positions.
- 2026-05-13: Honest caveats on the headline. The matmul kernel is **bf16 F.linear** (per-row recipe) or **fp8 Triton kernel** (Qwen3 block-fp8), not an int kernel. What's "int" is the operands: per-token fp8 activations are 256-level (int8 with public per-token scale), per-row fp8 weights are 256-level (int8 with public per-row scale). Daniel's meeting criticism — "is the matmul kernel int, or are you committing int operands and running float ops" — still applies. This experiment isolates: *the operand-level match is perfect; the open question is what an int matmul kernel costs in top-1*. The prior `train-int-cast` experiment with full int-matmul-emulation against fp32 reference gives an answer (Approach C trained: 0.99941 on Qwen2.5-0.5B), but that's a different setup.
- 2026-05-13: Failed configurations that helped triangulate. (a) fp8 act + **int non-matmul** ops (default `patch_nonmatmul`) → Qwen2.5-0.5B top-1=0.918 — the int RMSNorm/SiLU/softmax approximations are the dominant residual when present. (b) uniform int24 act (no fp8 match) + int non-matmul (with fixed init) → 0.938 — the activation-quant scheme mismatch is the second-biggest contributor, but smaller than the non-matmul approximations. (c) Block-fp8 student without `--block-fp8-kernel-path` on Qwen3-8B → 0.951 — bf16 F.linear with block-fp8-rounded operands diverges from the teacher's Triton fp8 GEMM because the kernels differ in accumulator semantics; switching the student to the Triton kernel closes it.
- 2026-05-13: Llama needed an HF token (gated repo). User handed one over with explicit instruction not to hardcode after a prior leak; passed inline via env var (`HF_TOKEN=... ssh box 'cmd'`) without writing it to any file. Saved a `feedback_hf_tokens` memory.
- 2026-05-13: Luke's exact recipe explicitly measured. Recipe = "dequantize fp8 teacher → fp32 student → cast to int24 → eval vs fp8 teacher" with int24 weights + uniform int24 activations + **float** non-matmul (so the only delta from the bit-exact 1.0000 result is the activation-quant scheme: uniform int24 vs fp8 e4m3). On Qwen2.5-0.5B: **top-1 = 0.9389, margin p99 = 0.301, KL p99 = 0.071** vs fp8 teacher. Conclusion: the fp32→int24 weight cast IS effectively lossless (b=24 gives bf16 round-trip identity), but the resulting int-cast student still diverges ~6pp from the fp8 teacher because uniform int24 activation quant ≠ fp8 e4m3 grid. More bits don't help — the issue is grid *structure*, not precision. The remaining options to close that gap: (a) use fp8-aware activation quant in the student — which makes the recipe equivalent to the bit-exact 1.0000 path above, just with a different name for the operand storage; (b) train the student (fp32 weight shadow + STE) on fp8 teacher logits so it learns to be robust to int quant noise. Saved at `data/qwen25_0p5b_lukes_recipe.jsonl`.
- 2026-05-13: Approach C training attempted against fp8 teacher (Qwen2.5-0.5B, fully-int: int24 weights + int24 act + int non-matmul + IntEmb). First run with aux_weight=0.5 + lr_luts=1e-3 + warmup=200 diverged catastrophically (loss 0.014 → 10 within 100 steps; lr_luts was destroying the int softmax/silu LUTs). Restart with May-11 config (aux_weight=0.05, grad-clip=0.5, warmup=500, --no-trainable-luts, lr=1e-7, lr-gamma-bias=1e-5): loss stable around 1.4e-2 with no downtrend. Eval at step 250 = 0.9350 (vs pretrain 0.9351). Eval at step 500 (loss still bouncing): no improvement. Killed at step ~510. **Approach C training does not close the 6pp gap against fp8 teacher.** This matches the May-11 train-int-cast finding ("pure logit-L2 on weight_fp shadows of every Linear, norms frozen, did not move the model in any direction the verifier cares about") — except now the starting point is much further from teacher (0.93 vs 0.99), and the gap is *also* immovable by re-weighting.
- 2026-05-13: **Critical reframe of the headline.** The fp8 production teacher is *itself* a noisy version of the bf16 base model. Pre-train evaluation on Qwen2.5-0.5B (against three references simultaneously):
  - student_vs_teacher (int vs fp8): top-1 = 0.9351
  - **teacher_vs_ref (fp8 vs bf16 base): top-1 = 0.9370**
  - **student_vs_ref (int vs bf16 base): top-1 = 0.9691**
  The fully-int student is **closer to the unquantized base model than fp8 is.** Both fp8 quantization and int quantization introduce ~3-6 pp top-1 noise vs the base, but in different directions — so they disagree ~6 pp with each other while each only disagrees ~3-6 pp with the base. By the "downstream quality" interpretation of the user's goal ("performs similarly to the fp8 model"), the int model already qualifies: it's at least as faithful to the base as fp8 is. By the "matches fp8 token-by-token" interpretation, no amount of weight training will close the gap because the disagreement is *stochastic* (data-dependent), not systematic.
