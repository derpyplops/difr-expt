# Experiment log: baseline-int-cast
Started 2026-05-11. See experiments/baseline-int-cast/plan.md.

- 2026-05-11: Scaffolding — created src/difr_expt with `int_cast.py` (IntLinear + per-row weight quant), `metrics.py` (top-1, top-5 overlap, logit L2, KL, Token-DiFR Gumbel margin per paper §4.2 Eq. 1), `run_baseline.py` (teacher-forced eval harness). Pinned torch/transformers to match the difr repo's pyproject. Smoke test on sshleifer/tiny-gpt2 (CPU) passed end-to-end. Two non-obvious bugs surfaced during testing: (1) quantize math at int32 magnitudes was crashing because float32 has ~256 spacing at 2^31, causing cast overflows — switched math to fp64 and kept scale at fp32 so bf16 source weights don't get a noisy scale; (2) Token-DiFR margin initially compared `cand.argmax(-1)` against the post-Gumbel verifier argmax, which gave nonzero margin even when ref==cand logits — fixed by also adding noise to cand before taking its argmax (matches paper's "provider's claimed token" semantics). 16 unit tests across int_cast and metrics, all green.
- 2026-05-11: Direction pivot — dropped the weights-only v1 in favor of full int conversion (weights + activations) per Daniel's call. Rewrote IntLinear with per-token dynamic activation quantization, capped effective bit width at 16 to keep int32×int32 → int64 accumulator from overflowing at Llama-3.1 8B's 14336-dim MLP (matches the constraint Luke hit in `int-model-approximation`). Default compute path is dequant-then-fp32-matmul, which equals true int matmul plus dequant in exact arithmetic and only differs by float reduction order — runs at normal GPU speed. Added a `true_int_matmul` flag for verification (CPU fallback à la Luke's repo, 2-3h per model). Tests updated; toy net hit 100% top-1 match.

Hit a torch-version snag: our pyproject pins torch==2.7.1 but the image ships 2.5.1+torchvision; pip-install bumped torch and left torchvision broken (`torchvision::nms does not exist`), which transformers' Qwen2 module triggers on import. Llama 3.1 8B comes out cleanest (top-1 0.9871, logit-L2 mean 11.6 vs Qwen3-8B's 22.7) — plausibly a tighter weight distribution. See reports/results-2026-05-11.md.
- 2026-05-11: Reconciliation with Luke's repo — Luke claimed ~99% top-1 by default, ours measured 97.9%. - 2026-05-11: Bit-width sweep on Qwen2.5-0.5B at {16, 20, 24, 28, 30} effective bits, every-position eval over 100 wikitext prompts. **All five runs give top-1 ≈ 0.977–0.980, logit-L2 mean glued at 20.36–20.39, KL p99 ~5.4e-3.** Bit width does not move the headline metric. Diagnostic: replaced every nn.Linear with a NON-quantized fp32-compute path (no quant at all, just W bf16 → fp32, matmul, → bf16) and re-ran; top-1 vs bf16 reference dropped to 0.9795. **The ~2pp floor is the difference between fp32-accumulated and bf16-native matmul reduction order, not quantization.** Adding more bits cannot close it.
- 2026-05-11: PTQ variants on top of the same sweep — added `quant_scheme` ∈ {symmetric, asymmetric, per_group_sym} to `int_cast.py` (`quantize_per_row_asym`, `quantize_per_token_asym`, `quantize_per_row_groupsym`, `quantize_per_token_groupsym`). Asymmetric uses per-row (max−min)/2qmax scale and zero-point; per-group splits the in-features axis (g=128 default; layers whose in_features % g ≠ 0 silently fall back to symmetric). New CLI flags `--quant-scheme`, `--group-size` in `run_baseline.py`. Ran symmetric/asymmetric/grp128/grp64 at 16-bit and asym/grp128 at 24-bit on Qwen2.5-0.5B/100-prompt. All variants land in the same 0.977–0.980 top-1 band — variation is within ±0.2pp noise. Consistent with the bit-width sweep diagnostic above: the floor is bf16-vs-fp32 reduction order, not the quant scheme. JSONs saved under `data/qwen25_0p5b_{asym16,grp128_16,grp64_b16,asymmetric_b24,asymmetric_b30,grp128_b24}.json`.

- 2026-05-11: Root-caused the 2pp floor. Added `--matmul-dtype {auto,fp32,bf16,fp16}` and `--no-quant` to `run_baseline.py`; routed dequantized operands through `F.linear` (bias fused inside cuBLAS addmm) instead of `x @ w.t() + b`. Both pieces are load-bearing: bf16 matmul *alone* (separate-add bias) still gave 0.979 top-1; bf16+F.linear matches `nn.Linear` bit-exact. A no-quant control gives top-1=1.0/l2=0.0 across all 15,348 wikitext positions — the floor was *bias-fusion order*, not reduction order. With this fix on Qwen2.5-0.5B (sym, w=a bits): 16→0.980, 24→0.981, 30→0.999, **31→0.9999** (target ≥99.9% hit, l2 0.06, kl_p99 5e-6). Asymmetric/per-group quant *hurts* (zero-point or group-boundary scales reintroduce noise). Added `activation_bits=0` sentinel for weight-only PTQ; **weight-only int31 + bf16 F.linear hits bit-exactness on all three models (Qwen2.5-0.5B / Qwen3-8B / Llama-3.1-8B all 1.0 / l2=0 / kl=0)**. Full w31a31: Qwen2.5-0.5B 0.9999, Qwen3-8B 0.9952, Llama-8B 0.9949 — activation outliers (absmax/median >1000x in MLP rows) compounding across 32+ layers and 14336-dim MLPs is the remaining gap on 8B. Tests 10/10 green. 31-bit accumulator overflows int64 over 14336-dim so this asymptote isn't ZK-realizable as-is; lesson is the float-equivalent baseline can be made *exact*, and any gap to a ZK target comes purely from the prover's bit budget.
- 2026-05-11: Sweep to find smallest int64-safe bit width hitting ≥99.9%. Ran Phase A (b∈{16,18,20,22,24,26,28} sym at w=a), Phase B (b=24 with per_group_sym g∈{32,64,128}, asymmetric, w20/a28 & w28/a20 mixed, skip-lm-head), Phase C (weight-only sweep w∈{24..31} a=0; w28+ a∈{18..22}; per-group weight-only g∈{8..128}; per-group both w&a g∈{4,8,16}; SmoothQuant α∈{0.5..0.9} +/- per-group +/- act-clip; activation outlier clip q∈{0.999, 0.9999}). Added `calibrate_smooth_scales`, `smooth_scale` buffer in IntLinear, `act_clip_quantile` argument to `quantize_per_token`, and the per_group_sym `skip_act` branch (the existing code crashed when `activation_bits=0` met `per_group_sym`). Pushed fp64 group-quant intermediate to CPU since `[O, ng, G]` at g=8 over 14336-dim OOM'd the H100. **Headline: no int64-safe config (max(w,a)≤24 per `2max(w,a)+log2(dim)≤63`) reaches 99.9% top-1 on the 8B models.** Best int64-safe full-quant: Qwen2.5-0.5B w=a=24 sym 0.9814; Qwen3-8B w=a=24 grp8 0.9866; Llama-3.1-8B w=a=24 sym 0.9864. Best weight-only (not ZK-realizable but instructive) at w=24 with grp16 weight quant: Qwen3-8B 0.9881, Llama 0.9897. Diagnostic key: bf16-mismatch fraction of `(x_int*scale).to(bf16)` vs original bf16 x is 0.0059/layer at b=24 sym w/ outlier-token activations — compounded over 32 layers gives the ~1.4% top-1 loss we observe. Per-group g=8 should drop this to ~0.0000 (synthetic test confirms) yet real-model top-1 stays at 0.987, suggesting another error pathway (likely cuBLAS bf16-tensor-core reduction order differing between deepcopy'd reference and the IntLinear stack). SmoothQuant (any α) and asymmetric did **not** help; act-clip catastrophically broke things (top-1<0.75) because legitimate outlier tokens get clipped away. Phase D verification: at b=24 on Qwen2.5-0.5B with `--true-int-matmul` (5 prompts, max_len=128), top-1=0.9803 — within 0.1pp of float-equiv 0.9814, confirming the int representation is faithful. **Conclusion: the ZK bit budget at int64 (max(w,a)≤24) is genuinely insufficient to hit 99.9% on the 8B models with the current bf16 reference pipeline.** Either widen the accumulator (int128 / Montgomery-friendly larger field), accept the 1.4% gap, or move the comparison off bf16 onto an fp32 reference where weight quant rounding could vanish. Tests 17/17 green. ~$2 of vast budget spent this round (well under the $20 cap).
