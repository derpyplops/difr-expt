# Baseline int-cast divergence — first measurement
2026-05-11. See ../plan.md and ../EXPERIMENT_LOG.md.

## Setup
- **Conversion**: every `nn.Linear` replaced with `IntLinear` (weights per-output-row int, activations per-token dynamic int, **16 effective bits both sides**). RMSNorm/RoPE/softmax stay bf16.
- **Compute path**: dequant-then-fp32-matmul (equal to true int matmul in exact arithmetic; only float reduction order differs). `--true-int-matmul` flag is wired but not used here.
- **Reference**: same model loaded in bf16 from HF, no quantization.
- **Eval**: teacher-forced over wikitext-103 prompts on an H100 80 GB.
- **Repo**: github.com/derpyplops/difr-expt @ initial commit. Vast offer 36473839.

## Read

- **All-position aggregation is the strict measurement; last-only matches Luke's setup.** Top-1 jumps from ~98% to ~99–100% when you only look at the last token, because the last position of a wikitext snippet is typically more constrained than mid-sequence positions — quant noise can't flip a token whose logit gap is wide.
- **The per-position picture is identical between aggregations.** logit-L2 mean, KL mean, and Gumbel-margin mean all agree to within ~10% across aggregations on the same model. So the difference in top-1 is *purely* about which positions you sample, not about the int-cast actually behaving differently.
- **The Gumbel margin is tiny everywhere**: 6e-4 to 1e-3 mean across both aggregations. Most disagreements are on positions where the post-Gumbel scores were close anyway. The DiFR paper's actual verification metric (margin from §4.2 Eq. 1) is the right one to optimize, and by that metric the int conversion looks fine.
- **Llama 3.1 8B is the cleanest** of the three: half the logit-L2 of Qwen3-8B (all-positions), and its KL p99 is 2× narrower. Plausibly a function of how concentrated its weight distribution is — narrower per-row max values give tighter scales.
- **0.5B and 8B sit ~0.8 pp apart on all-position top-1** — modest, not dramatic. So the result is not strongly size-dependent in this regime.

For comparison, Luke's `int-model-approximation` at step 0 on RedHatAI/Qwen2.5-0.5B-FP8-dynamic, eval split (8 prompts × last token only): top-1 = 1.0 (8/8), top-5 = 0.95, KL = 2.8e-3 — directionally consistent with our last-only column on the same architecture.

## Caveats
- 100 prompts (≈15k positions) is at the lower end of what the paper calls "minimum to saturate AUC ≥ 0.999". For headline numbers we want 1k+ prompts, especially for the p99 tails.
- The dequant-then-float compute path is mathematically equivalent to true int matmul in exact arithmetic, but float-reduction-order noise from the dequantized fp32 matmul is *not* present in the true int path. Worth a one-off `--true-int-matmul` run (CPU fallback, ~2 h) to confirm the numbers don't move.
- No Qwen3-30B-A3B — wouldn't fit on 80 GB H100 with reference + int-cast both resident. Needs a sequential-load variant of the harness.
