# Experiment log: train-nonmatmul-int
Started 2026-05-11. See experiments/train-nonmatmul-int/plan.md.

- 2026-05-11: Plan drafted. Strategy: keep b=24 IntLinear matmul implementation frozen (already bit-exact vs fp32 reference per prior experiment), replace RMSNorm / softmax / SiLU / attention matmuls with int-friendly approximations (Newton-Raphson invsqrt, lookup-table exp + sigmoid, int matmul), and train ONLY upstream compensation params (RMSNorm `γ`, Linear biases — total ~1M params for 8B). Pass criterion: top-1 ≥ 99.9% vs fp32 reference. Phase 1 (no training) is the most informative result — if approximation alone gets ≥99.9%, training is unnecessary. Estimated ~$45 compute across all four phases.
- 2026-05-11: Track A wrap — added unit tests for the int-RoPE implementation (`int_rope_apply` + `IntRopeApply` in `src/difr_expt/int_ops.py:449,545`, default `rope_bits=24` wired through `IntOpsConfig` in `patch_hf_model.py:58`). The op factors HF's `q*cos + rotate_half(q)*sin` so cos/sin share a single public scale `s_table = 1/(2^(b-1)-1)` (legal because |cos|,|sin| ≤ 1) and q/k use per-token symmetric quant — net cost is int24*int24 → int48 multiplies with an int49 sum, well inside int64. Two tests in `tests/test_int_ops.py`: `test_int_rope_close_to_float` builds realistic rotary cos/sin from inv_freq=1/10000^(i/half) (instead of random in [-1,1]) at shape [1,4,8,64] and gets max abs delta ~3.6e-7 vs float reference (threshold 1e-4); `test_int_rope_true_int_matches_float_equiv` confirms `true_int=True` int64-path matches the default float-equivalent path within ~1.2e-7 (threshold 1e-5). 40/40 tests passing, no implementation bugs surfaced.
- 2026-05-11: Phase 4 (Track C, literal-int validation on local CPU) — blocker found before any smoke run. Audited `src/difr_expt/int_ops.py` after the previous agent was killed at "model cached, about to run smoke". Of the four non-matmul ops, only two have a literal-int execution path implemented: `IntMatmul._true_int_forward` (int64 matmul with CUDA→CPU fallback, src/difr_expt/int_ops.py:398) and `int_rope_apply(..., true_int=True)` (int64 rotate_half + multiply-add, src/difr_expt/int_ops.py:506). The other three are float-equivalent only — `IntRMSNorm` carries a `true_int_path` class flag but its `forward()` has no `if self.true_int_path:` branch (lines 180–211 unconditionally take the float-equivalent path: quantize, fp64 sum-of-squares, NR invsqrt with float LUT, float multiply-out); `int_softmax` exposes no `true_int` parameter at all (the `e/s` step at line 290 uses float division, not the NR-reciprocal int circuit described in plan.md:209); `int_silu` likewise has no `true_int` knob, and `IntSiLUModule` in `patch_hf_model.py:77` is not in `set_true_int_path`'s isinstance tuple at src/difr_expt/int_ops.py:582. Net effect: invoking `run_baseline.py --true-int-nonmatmul` today flips IntMatmul + IntRopeApply to int64 but silently leaves RMSNorm/softmax/SiLU on the float path, so a "passing" run would not actually validate the integer-arithmetic forward — the very failure mode Phase 4 is designed to catch (plan.md:152). Per task instructions ("If any op's literal int path is missing, report that as the blocker and stop") did NOT run the smoke or 5-prompt validation; no JSON artifacts produced. Next step before resuming Phase 4: implement true-int branches for IntRMSNorm.forward (integer-domain NR invsqrt + int multiply-out — fp64 sum-of-squares already there), int_softmax (integer LUT entries, int64 sum, NR-reciprocal, int multiply), int_silu (integer sigmoid LUT, int multiply), then register IntSiLUModule in `set_true_int_path`.
- 2026-05-11: Phase 4 unblock — implemented the four missing literal-int execution paths flagged by the prior audit. (1) `IntRMSNorm._true_int_forward` (src/difr_expt/int_ops.py:213) added an `if self.true_int_path:` branch in `forward`: per-token symmetric quant → int64 sum-of-squares → public-scale dequant of variance → NR invsqrt seeded from the existing log-spaced LUT (CUDA-side `_invsqrt` runs on the model device via `.to(target_device).cpu()`) → int64 multiply `x_int * γ_int` (int24*int24=int48, safe in int64) → fp32 dequant by `x_scale * γ_scale * r`. (2) `int_softmax(..., true_int=True)` (int_ops.py:300) builds the exp LUT as int24 values under public scale `s_lut = 1/qmax`, picks them via the existing idx, int64-sums, and runs `_nr_reciprocal_int` — Newton step `r ← r*(2-s*r)` for 3 iterations from a float seed (2 was tight at ~5e-6, 3 gives ~2e-7, well under the 1e-4 test bar). (3) `int_silu(..., true_int=True)` (int_ops.py:401) per-token quant of x, int sigmoid LUT under `s_sig=1/qmax`, int48 product, fp32 dequant by `x_scale * s_sig`. (4) `set_true_int_path` (int_ops.py:633) now covers `IntSiLUModule` (lazy import to avoid circular) and propagates softmax true-int via a new `IntOpsConfig.true_int_nonmatmul` flag captured by the attention forward closure (patch_hf_model.py:155); the patcher attaches `attn_module._int_ops_cfg = cfg` so the walker can find it. The softmax remains a free function so cfg-threading is the cleanest path. Max abs deltas vs float-equivalent path: RMSNorm 4.8e-7, softmax 2.4e-7, SiLU 1.4e-6 — all ~ULP. Tests: added `test_int_rmsnorm_true_int_matches_float_equiv`, `test_int_softmax_true_int_matches_float_equiv`, `test_int_silu_true_int_matches_float_equiv`, plus `test_set_true_int_path_covers_silu_and_softmax_cfg` exercising the wiring. 44/44 pytest pass (40 prior + 4 new).
- 2026-05-11: IntEmbedding — closed the last forward-pass float gap by adding `IntEmbedding` to `src/difr_expt/int_cast.py` (per-vocab-row symmetric int24 quantization of the table, drop-in shape+dtype, handles `padding_idx`). The lookup itself is just indexing so there is no arithmetic to integerize; what changes is that the values returned are `int24 * public_per_row_scale`. Wired `patch_model_int_embedding(model, bits=24)` and exposed `--int-embedding`/`--embedding-bits` in `run_baseline.py`. Three new tests in `tests/test_int_cast.py` (close-to-float delta <1e-5 at b=24; padding_idx zero-out; patcher replaces nn.Embedding modules). 47/47 tests pass. With this, every op in the forward pass that has arithmetic content runs in integer arithmetic with public scales: Linear matmuls, RMSNorm, softmax, SiLU, attention Q@K.T and P@V, RoPE, and now the embedding table is a committed array of int24s. The only remaining floats are public-scale multiplies at dequant boundaries and residual additions (both fine for ZKP — known constants and exact under arbitrary-precision arithmetic).

- 2026-05-11: Phase 3 ablation sweep (Qwen2.5-0.5B, full-int + IntEmb, 100×512 wikitext, fp32 ref). Knob sweeps over softmax LUT, per-op opt-out, and HIPREC compositions, all at b=24 weights/activations. Best ablations: `no-softmax` (float softmax, rest int) → 0.99960 — isolates softmax LUT precision as the dominant remaining error; `smlut16k` → 0.99950 (vs 0.99928 default `smlut1k`+IntEmb baseline) — confirms LUT precision is the lever.
