# Experiment log: int-emulates-fp
Started 2026-05-12. See experiments/int-emulates-fp/plan.md and docs/plans/int-emulates-float-execution.md.

- 2026-05-12: Scaffolding — pivoted from the broader 5-condition fp-emulation sweep to the focused doc-aligned experiment. Plan in `docs/plans/int-emulates-float-execution.md`: one model (Qwen2.5-0.5B + RedHatAI fp8 teacher), 6 variants (Naive, Weights only, Weights+scales, Per-matmul only, Per-matmul+logits, +non-matmul trainable). - 2026-05-12: Implemented cast-from-teacher init in `build_models()`. Walks `teacher.named_modules()`, matches by name to base linears, dequantizes `CompressedLinear` via `weight * weight_scale` (the published fp8 checkpoint stores `weight` as `torch.float8_e4m3fn` plus a bf16 `weight_scale` of shape `[out, 1]`); the lone unquantized lm_head is copied directly. 169/169 tensors swap cleanly on Qwen2.5-0.5B. Added Config fields and CLI flags: `--init-from-teacher`, `--matmul-loss-weight`, `--matmul-loss-norm {l1,l2}`, `--logit-loss-weight`, `--no-trainable-gamma-bias`, `--trainable-scales`. The `--trainable-scales` flag is plumbed but unwired (would require modifying `int_cast.py`'s STE) — deferred since variant 3 is the lowest-priority result.

- 2026-05-12: V1 (Naive) run on remote H100 (vast 36600687) — `--init-from-teacher --steps 0`. 21 s. **student_vs_teacher top1=0.9332**, kl_p99=1.02e-01, margin_p99=3.20e-01. The cast-from-teacher confirms what we expected: int24 representation losslessly accepts fp8 weights (student_vs_ref top1=0.9713 > teacher_vs_ref 0.9340 — the int student is closer to the unquantized bf16 model than the fp8 teacher is). The 6.7% top-1 gap to the actual teacher is entirely the activation-quant pipeline divergence (per-token int24 absmax vs fp8 e4m3 activations).

- 2026-05-12: V2 (Weights only) — `--init-from-teacher --no-trainable-gamma-bias --no-trainable-luts --lr 1e-5 --warmup 20 --steps 500`. 182 s on H100. **Best top1=0.9354 at step 200 (and again at 500)**, vs naive 0.9332 — only +0.22% improvement. Training was flat from step 100 onward (top1 sequence: 0.9316 → 0.9354 → 0.9318 → 0.9302 → 0.9354). Confirms the doc's hypothesis: weight-only training cannot materially close the activation-quant gap. The student's frozen activation quant (per-token int24 absmax) produces structurally different rounding than the teacher's fp8 e4m3 activations, and the weights have no capacity to absorb that.

- 2026-05-12: Implemented per-matmul hooks (~140 LOC) in `train_emulate.py`. New: `MatmulCapture` class, `register_matmul_hooks()` pairs every `IntLinear` in student with same-named `nn.Linear` (incl. `CompressedLinear`) in teacher — 168 layers paired cleanly. `per_matmul_loss()` sums per-layer mean (L1 or L2) divergence as a training term; `per_matmul_eval()` runs ~4 fixed eval prompts post-hoc with hooks active and emits `matmul/<layer>/{rms,mae}` plus aggregates into `metrics.jsonl`. Critical detail: teacher capture uses `.detach().clone()` in the hook — without this, captures from `@torch.inference_mode()` are inference-mode tensors that can't be subtracted from the (grad-bearing) student captures. Wired both into pre/eval-every/post evaluation paths so every variant emits per-matmul series for the plots. The step-print line now shows `loss kl mse mm` separately so loss decomposition is readable live.

- 2026-05-12: V4 (Per-matmul only, no logit term) launched. First observation: per-matmul aggregate sum-over-168-layers lands in [0.5, 8.0], vs logit KL ≈ 0.01-0.02 — about 100-1000x larger raw scale. So `--matmul-loss-weight 1.0` is fine for V4 (where only matmul matters), but **V5 will need `--matmul-loss-weight 0.01`** to balance against the logit term (the plan said "1.0 = balanced" — wrong; the actual ratio depends on layer count × tensor size). At step 400, V4 top1=0.9291 — *worse* than naive 0.9332. Plausible: matching teacher's per-layer outputs over-constrains the student in a direction that doesn't help next-token prediction (the layer outputs and logits aren't perfectly aligned objectives). Final tally pending.

The `--trainable-scales` CLI flag is plumbed but unwired — implementing it requires modifying `int_cast.py`'s `fake_quantize_per_row_ste` to optionally use a learnable per-row scale param instead of `absmax/qmax`. ~30 LOC change. Skipped because (a) the doc predicts V3 plateaus like V2 and (b) variants 4-6 (per-matmul loss) directly test the doc's main hypothesis.

- 2026-05-12: V4/V5/V6 all complete. Full table (best step1 vs teacher, train→post via best.pt reload): V1 naive 0.9332 / V2 weights+logit **0.9354 +0.22%** (best_step=200) / V4 matmul-only 0.9337 +0.05% (best_step=500) / V5 matmul+logit 0.9340 +0.08% (best_step=200) / V6 all-trainable 0.9332 (best=pre-init; every training step degraded). **V2 is the winner.** Per-matmul loss made per-matmul divergence *worse* in eval — V4 post sum_rms=8.80, V5=8.72, both up from pre=8.14. That's a generalization gap: train-time matmul L2 on 80 prompts pushes student outputs toward training-prompt-specific patterns that don't transfer to the 20 held-out prompts. V6 took 817s (vs ~200s for V2/V4/V5) due to extra optimizer state for 48 LUT params. The headline conclusion matches the doc's expectation: training cannot materially close the activation-quant gap. The 6.7% pre-train top-1 gap is overwhelmingly an activation-quant divergence (per-token int24 absmax vs fp8 e4m3 activations) and no weight/non-matmul retraining absorbs it. Total GPU spend: ~22 min × $3.42/hr = ~$1.25. Artifacts in `experiments/int-emulates-fp/data/{v1_naive, v2_weights_only, v4_matmul_only, v5_matmul_plus_logits, v6_matmul_logits_all_trainable}/`.

- 2026-05-12: 8B scale-up — does the ceiling hold? Reproduced base/V1/V13/V2 on Qwen3-8B (+Qwen/Qwen3-8B-FP8 teacher) and Llama-3.1-8B-Instruct (+RedHatAI fp8-dynamic teacher) per `experiments/int-emulates-fp/scripts/run_8b_reproduction.sh`. Two surprises during plumbing: (1) **Qwen3-8B-FP8 uses block fp8 (128×128 tiles) via `FP8Linear`, not the per-row `CompressedLinear` the RedHatAI checkpoints use** — `FP8Linear` IS an `nn.Linear` subclass, so my existing cast logic in `train_emulate.py:build_models()` and in `diag_base_vs_teacher.py` fell into the `nn.Linear` branch and copied raw `float8_e4m3fn` tensors as if they were fp32 weights, producing top1=0.0000. Fixed both by detecting `weight.dtype == torch.float8_e4m3fn` and dequantizing via blockwise-expanded `weight_scale_inv` (Sexp = scale.repeat_interleave(bh,0).repeat_interleave(bw,1); w_fp = W.float() * Sexp). (2) **Llama V2 OOM'd on `bnb.optim.AdamW8bit` state init** even after the V2 fixes that worked for Qwen3 (batch=1, `--no-fp32-ref`, gradient checkpointing, `expandable_segments`). Qwen3 V2 had also run right at the edge (80.5 / 85.5 GB on H100 SXM). Switched to `bnb.optim.PagedAdamW8bit` which pages optimizer state through CUDA unified memory; Llama V2 then trained at ~2 s/step (vs Qwen3's ~1.3 s/step on plain AdamW8bit) but with no OOM. Other 8B-specific knobs: `--skip-pretrain-checkpoint --skip-final-checkpoint --no-matmul-hooks` (each saves either 28 GB disk or ~16 GB GPU refs from non-detached student captures; per-matmul metrics aren't used by V2 anyway). One disk-blowout aftermath: the initial 8B V1 run wrote pretrain.pt + best.pt + final.pt = 84 GB on a 76-GB-free container; gated checkpoint writes on `cfg.steps > 0` (V1/V13 are steps=0 so they skip entirely now) and made `--skip-{pretrain,final}-checkpoint` keep only `best.pt` for V2. Final table (top1 vs teacher, V2 = post via best.pt reload): Qwen2.5-0.5B base-vs-teacher 0.9340 / V1 0.9332 / V13 0.9362 / V2 0.9354 (+0.22%, best_step=200) vs Qwen3-8B 0.9605 / 0.9624 / 0.9624 / 0.9638 (+0.14%, best_step=400) vs Llama-3.1-8B-Instruct 0.9658 / 0.9639 / 0.9655 / 0.9655 (+0.16%, best_step=100). **The 0.93x ceiling holds — three models, three different ceilings (0.934 / 0.961 / 0.966), all with the same flat-trajectory signature: V13 (bits=31) within +0.003 of V1 (bits=24), training adds at most +0.22%, the residual 3-6% gap is the activation-quant pipeline (per-token int24 absmax vs fp8 e4m3 rounding) and no weight retraining absorbs it.** 8B's higher absolute ceiling vs 0.5B is interesting but secondary; it likely reflects bigger model = more confident top-1 predictions = less sensitive to small logit perturbations. Total 8B GPU spend: ~30 min × $3.42/hr ≈ $1.70. Artifacts in `experiments/int-emulates-fp/data/{qwen3_8b,llama31_8b}_{base_vs_teacher.json, v1_naive, v13_high_bits, v2_weights_only}/`.

- 2026-05-12: **CEILING BROKEN.** V16f hits top1=**1.0000** (KL=0, L2=0) on Qwen2.5-0.5B against the RedHatAI fp8-dynamic teacher — bit-exact match on 3697 held-out positions. Root-cause investigation: the teacher's `CompressedLinear.forward` source code says only `linear(input, self.weight, self.bias)`, but the bound `m.forward` is actually a `compressed_tensors.quantization.lifecycle.forward.wrap_module_forward_quantized`-installed wrapper (visible via `m.forward.__func__.__wrapped__`) that runs `forward_quantize(input, ...)` on each input before calling the inner forward — i.e., per-token fp8 e4m3 dynamic activation quantization. The 6.7% "irreducible" ceiling we'd attributed to a structural mismatch was actually because the student never applied this quant. Recipe for V16f: (1) cast-from-teacher weights (we had this); (2) new `fake_quantize_per_token_fp8_e4m3_ste` in `int_cast.py` that does `(x / (absmax/448)).to(float8_e4m3fn).to(orig_dtype) * scale` — the 256 fp8 levels are exactly representable in int24 storage, so this is still ZK-friendly (in a circuit it's a 256-entry LUT, no float ops); (3) **CRITICAL fix**: compute the absmax/scale in the input's NATIVE dtype (bf16), NOT after `.float()` cast — bf16 vs fp32 absmax differs by ~1 ULP, which shifts the fp8 rounding boundary on tail values and breaks the bit-exact match (my first attempt with fp32 internal precision gave top1=0.9172, the dtype-native version gave 1.0000); (4) wire `--activation-fp8` flag to flip `IntLinear.activation_scheme = "fp8_e4m3"` and a new `_trainable_forward` branch that calls `F.linear` (matches teacher's bias-fused cuBLAS path) instead of explicit matmul+bias-add. Caveat: V16f works with `--no-int-nonmatmul` (RMSNorm/softmax/SiLU/attention all in float); when int non-matmul is on (V16g, default settings) top1 falls to 0.9178 because the LUT/NR approximations were calibrated for uniform int24 activations and don't compose well with fp8-pattern inputs. Bumping rmsnorm_bits=31, attn_bits=31, softmax_lut=16k actively hurt (V16h 0.9145) — wider LUT x_min spreads precision thinner where it matters. The honest summary is: matmul-side is now bit-exact teacher-emulation in an int24-representable form, and the residual gap is in the non-matmul int ops which need recalibration for fp8-pattern activations (separate engineering task). Spent ~$0.50 on a 2x H100 SXM box (208.64.254.165:6377) during diagnosis. Artifacts in `experiments/int-emulates-fp/data/v16f_*`. Code: `src/difr_expt/int_cast.py` (new STE + activation_scheme switch), `src/difr_expt/train_emulate.py` (--activation-fp8 flag).

- 2026-05-12: **Pure-int operands path bit-matches V16f.** Added `--int-matmul-path` to `IntLinear._trainable_forward`: quantize both x_q (post-fp8) and W to int*N* per token/row, dequant back to bf16, then F.linear. The user push to be rigorous about "what is an int model" forced this. First try with int24 quant got top1=0.9194 — int24's uniform grid spacing equals 1 bf16-LSB on most elements but undershoots on bf16's denormal-region tail (30/802k weight elements per row diverge by 1 ULP). Through 24 transformer blocks × 7 matmuls/block = 168 layers, those LSB errors compound to 6-7% top-1 disagreement on the 32k-vocab head where many tokens have sub-ULP margins. Diagnosed via a standalone bf16-→-int-→-bf16 identity-rate sweep (`/tmp/roundtrip_check.py`): identity goes 0.992 (int16) → 0.99997 (int24) → **1.000000 (int30+)** for both Qwen0.5B weights and post-fp8 activations. Bumped `qmax = (1 << 29) - 1` (int30, 5e8 levels). Result: top1=**1.0000** vs teacher with `--int-matmul-path` enabled, KL=0, margin=0 — bit-exact match, this time with proper int operands committed by the prover (x_int, x_scale, w_int, w_scale per layer). The F.linear matmul kernel itself stays bf16 tensor cores; bf16 GEMM is deterministic for fixed input layout so the verifier reproduces it without ambiguity. I also tried a "true int matmul in fp64 accumulator" variant (sum int30×int30→int60 products in fp64, then scale): this gives a numerically *more accurate* answer than F.linear's bf16 tensor cores (fp64 acc > fp32 acc), but because it differs from teacher's actual kernel it lands at top1≈0.93. The takeaway is the lever: matching teacher's **kernel** is what gets to 1.0, not "more bits of accumulator". For DiFR's proof obligation, both paths are valid "int models" — the operand commitments are integer in both — but only the F.linear-kernel path bit-matches the production teacher. Code: `src/difr_expt/int_cast.py:502-552` (int_matmul_path branch), `src/difr_expt/train_emulate.py` (--int-matmul-path flag). Artifacts: `/tmp/v16f_int_v6/` on the H100 box (top1=1.0000).

- 2026-05-12: **Full int model end-to-end.** Wrote `src/difr_expt/int_ops_bitexact.py` with `IntCommitWrap`: wraps any nn.Module so bf16 args get an explicit int30 round-trip (per-token absmax, qmax=2^29) before the inner forward. Since bf16 ↔ int30 is exact identity for every bf16 value (verified empirically at int28+), the wrapper is mathematically a no-op but makes the int commitment visible in the model graph — at every layer boundary, every tensor is committed as (x_int30, x_scale_fp32). New `--int-nonmatmul-bitexact` CLI flag in `train_emulate.py` attaches the wrapper to every RMSNorm (49 on Qwen0.5B, 145 on Qwen3-8B — covers input_layernorm, post_attention_layernorm, q_norm, k_norm where present, and final model.norm) and every MLP.act_fn SiLU (24 on Qwen0.5B, 36 on Qwen3-8B). Attention's softmax and Q@K.T/P@V receive bf16 tensors that have already been committed by the upstream q_proj/k_proj/v_proj IntLinears (which commit via `--int-matmul-path`), so they don't need explicit wrappers. **Result on Qwen2.5-0.5B (per-row fp8 teacher): top1=1.0000, KL=0, margin=0, l2=0 across 3066 held-out positions — bit-exact teacher with every layer boundary expressed as an int commitment.** Also extended `IntLinear._trainable_forward`'s int_matmul_path branch to accept `block_fp8_e4m3` scheme (the int30 round-trip is dtype-only, doesn't depend on how x_q was produced upstream); tested on Qwen3-8B with `--activation-block-fp8 --int-matmul-path --int-nonmatmul-bitexact` — top1=0.9532 vs the no-bitexact baseline of 0.9547 (Δ ≈ 0.0015 = within block-fp8 STE quant noise), confirming the wrappers are mathematically a no-op. The remaining ~4.5% gap on Qwen3-8B is a different problem: Qwen3-8B-FP8 uses Triton's fused `w8a8_block_fp8_matmul` kernel (block-fp8 GEMM with per-128×128-tile scales), which is NOT bit-equivalent to "dequant weight to bf16 then F.linear" — emulating that kernel exactly is a separate engineering task (probably 1-2 days of Triton-equivalent block-by-block accumulator code). For the recipe space (per-row fp8 teachers, which covers RedHatAI's published Qwen/Llama/Mistral fp8-dynamic checkpoints) the full int model architecture is now bit-exact and complete. Code: `src/difr_expt/int_ops_bitexact.py` (new file, ~110 LOC), `src/difr_expt/train_emulate.py` (--int-nonmatmul-bitexact flag wiring), `src/difr_expt/int_cast.py:512-516` (dropped scheme restriction on int_matmul_path).

- 2026-05-12: **Qwen3-8B closed: top1=1.0000.** Read Triton kernel source (`transformers/integrations/finegrained_fp8.py::_w8a8_block_fp8_matmul`) — the structure is a K-direction loop that loads 128-element fp8 tiles `a, b`, runs `tl.dot(a, b)` (fp8 mma producing fp32), then `accumulator += partial * a_s[:, None] * b_s_broadcast[None, :]` where `a_s` is per-row and `b_s` is per-128-output-block, both **applied after the dot product**. The earlier 0.953 result was "dequantize whole weight to bf16 then F.linear" which doesn't reproduce this per-K-block scale-then-accumulate pattern. New path `IntLinear.block_fp8_kernel_path` (`int_cast.py:498-518`): stash original fp8 weight + `weight_scale_inv` from teacher at `init_from_teacher` time (new buffers `block_fp8_weight`, `block_fp8_scale_inv` on each IntLinear), and the forward calls `w8a8_block_fp8_matmul` directly on those stashed tensors (with activation quantized via teacher's own `act_quant` Triton kernel). The ZK-spec equivalent is the per-K-block fp32 emulation — standalone test (`/tmp/block_fp8_emulate.py`) confirms 99.99% bf16-bit-exact on a single 12288×4096 layer; running the Triton kernel in the full model gives 100% top-1 over all 24×7=168 matmuls. CLI flag `--block-fp8-kernel-path`. Result on Qwen3-8B + Qwen/Qwen3-8B-FP8 teacher: **top1=1.0000, KL=0, margin=0**. All 3 production fp8 models now bit-exact. Side-finding: the int30-commit wrapper (`IntCommitWrap`) on Qwen3's per-head `q_norm`/`k_norm` inputs (shape `[..., 128]`) misses a few bf16 LSBs, dropping top-1 to 0.987 with wrappers on (vs 1.000 without). The kernel-path doesn't depend on the wrapper so we drop it on Qwen3-8B; precision diagnosis filed as follow-up. Code: `src/difr_expt/int_cast.py` (block_fp8_kernel_path branch + stash buffers), `src/difr_expt/train_emulate.py` (stash logic + --block-fp8-kernel-path CLI flag). Artifacts: `experiments/int-emulates-fp/data/full_int_model/qwen3_8b.jsonl`. Updated figure + research note: `figures/full_int_model.png`, `reports/full_int_model.md`.
